## ï¿½ Shared ALB Configuration

All three Elastic Beanstalk environments share a single Application Load Balancer, configured with path-based routing rules managed by each environment.

### Listener Rules Configuration

| Rule Name           | Path Patterns                                                  | Target Service     | Priority | Port |
| ------------------- | -------------------------------------------------------------- | ------------------ | -------- | ---- |
| `inspectionapirule` | `/api/inspections`, `/api/inspections/*`, `/api/presigned-url` | Inspection API     | 100      | 3001 |
| `reportservicerule` | `/api/reports`, `/api/reports/*`                               | Report Service     | 110      | 3002 |
| `frontendrule`      | `/*`                                                           | Frontend           | 999      | 8080 |
| Default             | No match                                                       | 404 Fixed Response | -        | -    |

### Benefits

1. **Cost Optimization**: Single ALB ($16-20/month) instead of three ($48-60/month)
2. **Simplified DNS**: One domain/URL for the entire application
3. **Direct Routing**: ALB routes API requests directly to backend services
4. **No Proxy Overhead**: Frontend doesn't need proxy middleware
5. **Unified SSL**: Single certificate for all services

### Terraform Configuration

```hcl
# Each EB environment connects to the shared ALB
setting {
  namespace = "aws:elasticbeanstalk:environment"
  name      = "LoadBalancerIsShared"
  value     = "true"
}
setting {
  namespace = "aws:elbv2:loadbalancer"
  name      = "SharedLoadBalancer"
  value     = aws_lb.main.arn
}
setting {
  namespace = "aws:elbv2:listenerrule:<rulename>"
  name      = "PathPatterns"
  value     = "/api/path/*"
}
```

## ğŸ¯ Design Decisions

### 1. Why Elastic Beanstalk?

| Decision                   | Rationale                                                            |
| -------------------------- | -------------------------------------------------------------------- |
| **Managed Platform**       | Reduces operational overhead for deployment, scaling, and monitoring |
| **Auto Scaling**           | Handles traffic spikes without manual intervention                   |
| **Blue/Green Deploy**      | Zero-downtime deployments via Immutable policy                       |
| **CloudWatch Integration** | Built-in logging and metrics                                         |

### 2. Why 3-Tier Architecture?

| Tier             | Benefit                                                  |
| ---------------- | -------------------------------------------------------- |
| **Presentation** | Separate frontend allows independent scaling and updates |
| **Application**  | Microservices enable isolated development and deployment |
| **Data**         | Centralized data with read replicas for performance      |

### 3. Why Read Replica?

| Use Case              | Benefit                                         |
| --------------------- | ----------------------------------------------- |
| **Report Generation** | Offloads read-heavy queries from primary        |
| **Analytics**         | Allows complex queries without affecting writes |
| **High Availability** | Can be promoted to primary if needed            |

### 4. Why Presigned URLs for S3?

| Benefit         | Description                          |
| --------------- | ------------------------------------ |
| **Security**    | No direct S3 access from client      |
| **Performance** | Direct upload to S3, bypasses server |
| **Scalability** | Reduces server load for file uploads |
